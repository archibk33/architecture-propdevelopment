# Space Archiboard - Архитектурный анализ PropDevelopment

## Описание проекта

Данный проект представляет собой комплексный анализ архитектуры и безопасности для компании PropDevelopment - крупной строительной компании, входящей в топ-3 застройщиков страны. Проект включает пять основных задач по обеспечению безопасности и управлению доступом в облачной инфраструктуре.

## Структура проекта

### Task1 - Анализ безопасности данных
- **Цель**: Классификация данных по стандартам ISO/IEC 27001/27002
- **Результат**: Mindmap диаграмма с категориями данных и оценкой рисков
- **Файлы**: `data_analysis.md`, `security_mindmap.drawio`, `README.md`

### Task2 - Проверочный лист безопасности
- **Цель**: Создание комплексного проверочного листа для бизнес-систем
- **Результат**: 30-пунктный лист с рекомендациями по безопасности
- **Файлы**: `security_checklist.md`, `architecture_analysis.md`, `security_recommendations.md`, `README.md`

### Task3 - Внешние интеграции "Умный дом"
- **Цель**: Проектирование интеграции с сервисами "Умный дом"
- **Результат**: C4 диаграммы и требования к безопасности интеграций
- **Файлы**: `smart_home_requirements.md`, `context_diagram.drawio`, `PropDevelopment_С4_model.drawio`

### Task4 - Ролевой доступ Kubernetes
- **Цель**: Настройка ролевого доступа к Kubernetes кластеру
- **Результат**: Скрипты создания пользователей, ролей и привязок для Windows и Linux
- **Файлы**: 
  - Windows: `create_users_simple.bat`, `create_roles.bat`, `bind_users_to_roles.bat`, `test_rbac.bat`, `cleanup.bat`, `README.md`
  - Linux: `create_users.sh`, `create_roles.sh`, `bind_users_to_roles.sh`, `README.md`
  - Общие: `RBAC.md`, `roles_table.md`, `setup_instructions.md`

### Task5 - Сетевые политики Kubernetes
- **Цель**: Разграничение трафика между сервисами в кластере
- **Результат**: NetworkPolicies для изоляции пользовательских и административных сервисов
- **Файлы**: `network_policies.yaml`, `cleanup.bat`, `README.md`

## Ключевые выводы по безопасности

### Критические проблемы выявлены:
1. **Множественные точки аутентификации** - отсутствие централизованного управления доступом
2. **Утечка персональных данных** - нарушение 152-ФЗ, риски до 6 млн рублей штрафов
3. **Недостаточная защита API партнеров** - отсутствие единых политик безопасности
4. **Отсутствие мониторинга** - нет централизованного наблюдения за безопасностью

### Рекомендации по устранению:
1. **Внедрение API Gateway** - централизованный контроль доступа к API
2. **Централизация аутентификации** - единый Identity Provider
3. **Сетевая сегментация** - создание DMZ и изоляция критических систем
4. **Внедрение SIEM** - мониторинг и обнаружение инцидентов

## Техническая архитектура

### Организационная структура:
- **4 домена**: Продажи, ЖКУ, Финансы, Данные
- **25 пользователей**: Администраторы, DevOps, Разработчики, Менеджеры
- **27 ролей**: Кластерные и доменные роли с принципом наименьших привилегий

### Kubernetes кластер:
- **4 namespace**: Изоляция по бизнес-доменам
- **34 привязки**: RoleBindings, ClusterRoleBindings, ServiceAccounts
- **4 NetworkPolicies**: Сетевая изоляция между пользовательскими и административными сервисами

## Запуск проекта

### Предварительные требования:
- Docker Desktop с включенным Kubernetes
- kubectl для работы с кластером

### Порядок выполнения Task4 (Windows):
```bash
cd Task4/windows
create_users_simple.bat
create_roles.bat
bind_users_to_roles.bat
test_rbac.bat
```

### Порядок выполнения Task4 (Linux):
```bash
cd Task4/linux
chmod +x *.sh
./create_users.sh
./create_roles.sh
./bind_users_to_roles.sh
```

### Порядок выполнения Task5:
```bash
cd Task5
# Создание сервисов
kubectl run front-end-app --image=nginx --labels role=front-end --expose --port 80
kubectl run back-end-api-app --image=nginx --labels role=back-end-api --expose --port 80
kubectl run admin-front-end-app --image=nginx --labels role=admin-front-end --expose --port 80
kubectl run admin-back-end-api-app --image=nginx --labels role=admin-back-end-api --expose --port 80

# Применение сетевых политик
kubectl apply -f network_policies.yaml
```

## Соответствие стандартам

- **ISO/IEC 27001/27002** - классификация данных и управление рисками
- **152-ФЗ "О персональных данных"** - защита персональных данных
- **GDPR** - требования к обработке данных
- **Принцип наименьших привилегий** - минимально необходимые права доступа
- **Сетевая сегментация** - изоляция критических систем

## Бюджетная оценка (ДОПОЛНИТЕЛЬНЫЕ РАЗМЫШЛЕНИЯ)

**Общий бюджет на первый год**: 5-7 млн рублей
- Лицензии на ПО безопасности: 2-3 млн руб/год
- Инфраструктура: 1-1.5 млн руб/год
- Персонал: 3-4 специалиста по безопасности
- Консультации: 500-800 тыс руб

## Ожидаемые результаты

- Снижение рисков безопасности на 80%
- Сокращение времени обнаружения инцидентов на 70%
- Соответствие требованиям регуляторов
- Повышение доверия клиентов
- Полная изоляция пользовательских и административных доменов
- Централизованное управление доступом в Kubernetes кластере

## Заключение

Проект Space Archiboard успешно реализован и включает все необходимые компоненты для обеспечения безопасности инфраструктуры PropDevelopment. Достигнута полная изоляция между доменами, настроен ролевой доступ в Kubernetes кластере и реализованы сетевые политики для контроля трафика между сервисами.

Система готова к использованию в продакшн среде с дополнительной настройкой мониторинга и логирования.
