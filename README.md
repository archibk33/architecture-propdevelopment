# Space Archiboard - Архитектурный анализ PropDevelopment

## Описание проекта

Данный проект представляет собой комплексный анализ архитектуры и безопасности для компании PropDevelopment - крупной строительной компании, входящей в топ-3 застройщиков страны.

## Структура проекта

### Task1 - Анализ безопасности данных
- **Цель**: Классификация данных по стандартам ISO/IEC 27001/27002
- **Результат**: Mindmap диаграмма с категориями данных и оценкой рисков
- **Файлы**: `data_analysis.md`, `security_mindmap.drawio`, `README.md`

### Task2 - Проверочный лист безопасности
- **Цель**: Создание комплексного проверочного листа для бизнес-систем
- **Результат**: 30-пунктный лист с рекомендациями по безопасности
- **Файлы**: `security_checklist.md`, `architecture_analysis.md`, `security_recommendations.md`, `README.md`

### Task3 - Внешние интеграции "Умный дом"
- **Цель**: Проектирование интеграции с сервисами "Умный дом"
- **Результат**: C4 диаграммы и требования к безопасности интеграций
- **Файлы**: `smart_home_requirements.md`, `context_diagram.drawio`, `container_diagram.drawio`

### Task4 - Ролевой доступ Kubernetes
- **Цель**: Настройка ролевого доступа к Kubernetes кластеру
- **Результат**: Скрипты создания пользователей, ролей и привязок
- **Файлы**: `roles_table.md`, `create_users.sh`, `create_roles.sh`, `bind_users_to_roles.sh`, `setup_instructions.md`

### Task5 - Сетевые политики Kubernetes
- **Цель**: Разграничение трафика между сервисами в кластере
- **Результат**: NetworkPolicies для изоляции сервисов
- **Статус**: В разработке

## Ключевые выводы по безопасности

### Критические проблемы выявлены:
1. **Множественные точки аутентификации** - отсутствие централизованного управления доступом
2. **Утечка персональных данных** - нарушение 152-ФЗ, риски до 6 млн рублей штрафов
3. **Недостаточная защита API партнеров** - отсутствие единых политик безопасности
4. **Отсутствие мониторинга** - нет централизованного наблюдения за безопасностью

### Рекомендации по устранению:
1. **Внедрение API Gateway** - централизованный контроль доступа к API
2. **Централизация аутентификации** - единый Identity Provider
3. **Сетевая сегментация** - создание DMZ и изоляция критических систем
4. **Внедрение SIEM** - мониторинг и обнаружение инцидентов

## Техническая архитектура

### Организационная структура:
- **4 домена**: Продажи, ЖКУ, Финансы, Данные
- **25 пользователей**: Администраторы, DevOps, Разработчики, Менеджеры
- **27 ролей**: Кластерные и доменные роли с принципом наименьших привилегий

### Kubernetes кластер:
- **4 namespace**: Изоляция по бизнес-доменам
- **34 привязки**: RoleBindings, ClusterRoleBindings, ServiceAccounts
- **4 NetworkPolicies**: Сетевая изоляция между доменами

## Запуск проекта

### Предварительные требования:
- Docker Desktop с включенным Kubernetes
- OpenSSL для создания сертификатов
- kubectl для работы с кластером

### Порядок выполнения Task4:
```bash
cd Task4
chmod +x *.sh
./create_users.sh
./create_roles.sh
./bind_users_to_roles.sh
```
